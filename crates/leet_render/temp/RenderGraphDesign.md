# LEET Render Graph Design

This file is the design document for the LEET render graph work.

The immediate goal is not to copy every RED renderer implementation detail
literally. The goal is to preserve the allocator contract and data model clearly
enough that each layer can be implemented without painting LEET into a smaller
dead-end shape.

The first layer documented here is the frame resource allocator, because it is
the part most likely to be confused with asset loading, Bevy texture storage, or
Vulkan-style descriptor sets.

---

## 1. Frame Resource Allocator

### Allocator Contract Summary

The frame resource allocator is a RED-style frame-local resource planner adapted
to Rust, Bevy, and wgpu. It records logical resource requests, validates that the
same request stream is replayed during consume, resolves lifetimes, and assigns
pooled texture or buffer resources.

Locked contract:

- node code talks to `rctx`; allocator-facing methods are implementation hooks
- logical tags are compact, scoped by render-flow space, and cheap to copy
- tags are not raw strings, debug labels, or direct GPU handles
- resources are declared with `FrameResourceDesc::Texture` or
  `FrameResourceDesc::Buffer`
- declaration records intent; typed getters retrieve only after resolve
- `PreConsume`, `Resolve`, `Consume`, and `Cleanup` are distinct Rust phases
- pre-consume and consume request streams must match in order and meaning
- lifetimes are request-time based, so a tag can point to different resources at
  different points in the frame
- the `FrameResourcePool` owns reusable texture and buffer handles
- imported resources are tracked from day one for both textures and buffers, but
  the allocator must not recycle or destroy them
- LEET/wgpu does not expose physical memory aliasing; V1 still performs
  lifetime-aware same-frame and cross-frame pool reuse
- the internal model must keep the hooks needed for stricter backend placement,
  ownership rules, queue lifetime extension, and possible physical aliasing on a
  future non-wgpu backend

RED source references in this document are implementation anchors. They explain
which behavior LEET is preserving and where the Rust design intentionally adapts
RED-specific details such as unions, ESRAM flags, raw pointers, and physical
aliasing APIs.

### Resource Tags And Flow Spaces

This mirrors RED's `renderFlowNameTag.h`.

The allocator does not use plain strings as resource keys. It uses compact
logical tags. A tag is built from:

- resource name hash
- render-flow space
- optional autogenerated id

The name is the human/logical resource name:

```text
"color"
"depth"
"gbuffer0"
"velocityBuffer"
"SSAO_linearDepth"
```

The render-flow space scopes the name. This is how the same graph can render
multiple cameras without their resources colliding.

```text
"color" + flow space 1 -> camera A color
"color" + flow space 2 -> camera B color
"color" + flow space 0 -> shared/global color
```

Unique/global nodes use the shared flow space. Camera/view nodes use the flow
space assigned to their camera data.

Rust target shape:

```rust
pub struct RenderFlowName {
    hash: u32,
    debug_name: Option<&'static str>,
}

pub struct RenderFlowNameTag {
    hash: u32,
    flow_space: RenderFlowSpace,
    auto_id: RenderFlowAutoId,
}

pub struct RenderFlowSpace(u8);
```

The RED bit layout is:

```text
24 autogenerated-id bits
8 flow-space bits
```

RED combines these with the name hash:

```text
tag_hash = name_hash ^ ((flow_space << 24) | auto_id)
```

LEET should preserve this compact-key behavior unless there is a strong reason
to change it. The key is intentionally cheap to copy, hash, compare, and sort.
It is not a string identity.

The hash remains the allocator identity. LEET also stores the semantic flow
space and autogenerated id next to the hash for diagnostics, validation, and
debug dumps. Equality, ordering, and map keys are still based on the compact
hash identity, not on debug strings.

Node-facing helper:

```rust
let color = rctx.rt_name_tag("color");
```

This helper only creates a logical tag. It does not allocate a resource.

`rctx` means the render node execution context. It is the node-facing API,
similar in role to RED's `SRenderNodeImplContext`.

`allocator` means the frame resource allocator behind `rctx`, similar in role to
RED's `CRenderFlowResourceAllocator`.

Nodes should normally talk to `rctx`, not directly to the allocator:

```text
render node
  -> rctx
      -> allocator
          -> FrameResourcePool
```

`rctx` owns the context-sensitive decisions:

- current camera
- current render-flow space
- whether the current node is unique/global
- current resource phase
- current render-flow group

The allocator owns the request stream, lifetime analysis, and backing resource
assignment.

`rt_name_tag` automatically chooses the correct flow space for the current node:

```text
camera/view node
  rctx.rt_name_tag("color") -> "color" in that camera's flow space

unique/global node
  rctx.rt_name_tag("color") -> "color" in shared flow space
```

The node should not usually request shared/global scope explicitly. That is a
property of the node context.

Allocation and access are separate operations:

```text
rctx.rt_name_tag("color")
  -> create only the logical tag in the current flow space

rctx.declare_resource(color_tag, FrameResourceDesc::Texture(texture_desc))
  -> record that the logical tag needs a texture with this descriptor

rctx.declare_resource(buffer_tag, FrameResourceDesc::Buffer(buffer_desc))
  -> record that the logical tag needs a buffer with this descriptor

rctx.get_texture(color_tag)
  -> retrieve the resolved GPU texture during consume

rctx.get_buffer(buffer_tag)
  -> retrieve the resolved GPU buffer during consume
```

`declare_resource` does not create GPU resources immediately. It records intent
in the allocator request stream. The `FrameResourceDesc` variant makes the
resource kind explicit at the call site. Actual texture/buffer backing assignment
happens only during the allocator `Resolve` phase, after pre-consume has recorded
the full request stream and before consume tries to retrieve resources.

The node-facing declaration API should return `Result<()>`. The tag is already
known, and the descriptor is an input. The resolved resource is retrieved later
with a typed getter.

```text
PreConsume
  rctx.declare_resource(color_tag, FrameResourceDesc::Texture(texture_desc))
  rctx.use_begin(color_tag, ResourceUsage::WRITE)
  rctx.use_end(color_tag)

Resolve
  allocator computes lifetimes
  allocator creates/reuses backing textures and buffers
  allocator assigns logical tags to backing resources

Consume
  rctx.declare_resource(color_tag, FrameResourceDesc::Texture(texture_desc))
  rctx.use_begin(color_tag, ResourceUsage::WRITE)
  rctx.get_texture(color_tag)
  rctx.use_end(color_tag)
```

Important invariants:

- tag creation must automatically use the node's current flow space
- unique/global nodes must use shared flow space
- camera/view nodes must use their assigned camera flow space
- dynamic names are allowed, but static names should be cheap
- invalid/null tags must exist for optional resources
- nameless allocation requests need an autogenerated id path; RED generates the
  id at request-recording time from the current flow group and request position
- debug names are for diagnostics, not allocator identity

Node-facing examples should use `rctx`:

```rust
let color = rctx.rt_name_tag("color");
rctx.declare_resource(color, FrameResourceDesc::Texture(color_desc))?;
rctx.use_begin(color, ResourceUsage::WRITE)?;

let clustered_lights = rctx.rt_name_tag("clustered_lights");
rctx.declare_resource(
    clustered_lights,
    FrameResourceDesc::Buffer(clustered_lights_desc),
)?;
```

Allocator-facing examples should be reserved for allocator implementation notes:

```rust
allocator.request_declare_resource(tag, flow_group, desc);
allocator.request_use_begin(tag, flow_group, usage);
```

### Node Context Convenience Layer

This mirrors the useful parts of RED's `SRenderNodeImplContext` without making
nodes depend on allocator internals.

In Rust this layer is `render_node_impl_context.rs`, not part of the
`resources/` module. The allocator remains the lower-level request/lifetime/pool
engine; the implementation context owns node-local flow group, flow space,
unique-node behavior, temp tag generation, and node-facing RT helper syntax.

`rctx` should provide convenience helpers for common node code, but those helpers
must still record the same underlying request stream:

- `rctx.rt_name_tag(name)` creates a tag in the current node flow space
- optional shared-name helpers may exist for rare explicit shared/global cases
- temp allocation helpers are nameless allocation requests with an autogenerated
  id path, not a separate allocator model
- descriptor helpers may fill color/depth/velocity/composition descriptors from
  the current camera or render context
- optional getters may return `None` for invalid/null optional tags

Temp helpers are for scratch identity, not immediate memory release. A temp
resource is still a normal transient frame resource:

```rust
let scratch = rctx.temp_resource_tag("ssao_scratch");
rctx.declare_resource(scratch, FrameResourceDesc::Texture(scratch_desc))?;
```

The debug name helps diagnostics. The allocator identity comes from the
autogenerated request-position id. The resource lifetime still comes from
`declare_resource`, `use_begin`, `use_end`, `free`, and other request-stream
events, and the resolved allocation may still come from the `FrameResourcePool`.

RED exposes an RAII `ScopeNameTag`, but LEET intentionally keeps the Rust
node-facing API explicit. Nodes should spell out the request stream they are
recording:

```rust
rctx.use_begin(color, ResourceUsage::WRITE)?;
let texture = rctx.get_texture(color)?;
rctx.use_end(color)?;
```

This avoids a second compact API path that hides the allocator semantics. During
`PreConsume`, `use_begin` and `use_end` record the use range. During `Consume`,
the same calls replay and validate the range, and typed getters are valid only
after resolve.

Implementation choice still open:

- Either preserve RED's exact XOR hash layout, or use a stronger internal key
  while preserving the same semantic fields. If this changes, the allocator
  still needs compact `Copy` keys and explicit flow-space scoping.

### Allocator Public API

This mirrors RED's `renderFlowResourceAllocator.h` at the behavior level.

The public allocator API is a request-recording interface. Nodes do not create
GPU resources directly through it. They issue logical resource operations in
graph execution order, and the allocator records those operations for later
resolution.

RED's public request surface includes:

- allocate by descriptor
- allocate by another tag's descriptor
- inject/import an external texture
- query whether a tag was allocated
- use begin
- use end
- free
- swap two logical tags
- swap a logical tag with an external texture or buffer reference
- begin queue
- end queue
- queue sync
- decision
- get resolved texture
- get resolved buffer

The Rust node-facing shape should stay close to this behavior, but use explicit
resource-kind enums instead of RED's packed C++ descriptor union:

```rust
let color = rctx.rt_name_tag("color");
rctx.declare_resource(color, FrameResourceDesc::Texture(color_desc))?;
rctx.use_begin(color, ResourceUsage::WRITE)?;
rctx.use_end(color)?;

let lights = rctx.rt_name_tag("clustered_lights");
rctx.declare_resource(lights, FrameResourceDesc::Buffer(buffer_desc))?;
```

Declaration by another tag's descriptor should be a distinct request operation,
not a fake descriptor variant:

```rust
let color = rctx.rt_name_tag("color");
let temp = rctx.rt_name_tag("post_temp");

rctx.declare_resource(color, FrameResourceDesc::Texture(color_desc))?;
rctx.declare_resource_like(temp, color)?;
```

`declare_resource_like(dst, src)` records that `dst` is a separate logical
resource whose kind and descriptor are copied from `src`. The source tag must
already be declared or imported in the same replay path. This keeps
`FrameResourceDesc` as a concrete texture/buffer creation description, while
preserving RED's allocate-by-tag behavior.

The destination may be a nameless/autogenerated request tag, but the source must
be an explicit valid tag. RED asserts that the descriptor source is not the
nameless-allocation sentinel.

The allocator-facing shape is the lower-level version:

```rust
allocator.request_declare_resource(tag, flow_group, desc)?;
allocator.request_declare_resource_like(dst_tag, src_tag, flow_group)?;
allocator.request_use_begin(tag, flow_group, usage)?;
allocator.request_use_end(tag, flow_group)?;
```

The typed getters stay separate because retrieval is only valid after resolve:

```rust
let texture = rctx.get_texture(color)?;
let buffer = rctx.get_buffer(lights)?;
```

Requesting allocation and retrieving a resource must never be the same
operation. Declaration records intent. Resolve assigns backing resources. Getters
only read the resolved assignment.

Querying whether a resource is declared is also a recorded request, equivalent
to RED's `RequestIsAlloc`:

```rust
if rctx.is_declared(depth)? {
    rctx.use_begin(depth, ResourceUsage::READ)?;
    // During Consume, this is where `rctx.get_texture(depth)?` is valid.
    // During PreConsume, the same branch only records the use range.
    rctx.use_end(depth)?;
}
```

`is_declared` must not be implemented as an informal map lookup. If it affects
branching, then `PreConsume` and `Consume` must call it at the same point in the
request stream, and the allocator must assert/error if replay diverges.

RED treats allocation requests as successful request-stream declarations.
`is_declared` exists as a communication tool between nodes and as a replayed
request; it is not a late GPU allocation probe.

LEET should not copy RED's unsafe pre-consume `RequestIsAlloc` behavior blindly.
In RED, pre-consume `RequestIsAlloc` effectively returns `true` by default, while
consume checks resolved tag lifetimes. LEET should make `is_declared`
deterministic across phases:

```text
PreConsume
  record/query a graph-known declaration state

Consume
  replay the same query and return the same recorded result
```

If a branch is runtime-variable, use `rctx.decision(...)`. If a resource is
optional, use an invalid/null tag and an optional getter. `is_declared` must not
return `true` in pre-consume and `false` in consume; that should assert/error as
request-stream divergence.

Request replay matching is exact enough to prove the same logical resource flow,
but RED has deliberate exceptions for values that are meant to be stabilized by
the allocator:

- `Decision` does not require the incoming `baseDecision` to match; consume
  returns the decision recorded during pre-consume
- `SwapWithExternal` validates the tag, group, order, and request kind, but not
  the raw external resource handle
- import/injection validates the logical request identity, not pointer equality
  of the imported GPU object
- `QueueSync` is validated by sync type; the flow group is supplied by the
  request stream context

LEET should keep this as explicit request equality logic. It should not compare
opaque GPU handles when the API contract is "same request at the same point",
and it should not ignore logical fields such as tag, flow group, descriptor,
usage flags, queue type, or request order.

Rust should not derive replay equality blindly for `ResourceRequest`. RED's
`SRequest::operator==` is hand-written per request type. LEET should use an
explicit replay matcher:

```rust
impl ResourceRequest {
    pub fn matches_replay(&self, recorded: &Self) -> bool;
}
```

The request stream should stay compact. RED packs requests into a small union
because there can be hundreds or thousands of requests per frame. Rust should
prefer enum payloads made of compact ids, descriptors, and handles over heap
strings or large debug payloads in every request.

Swapping with an external resource is an in/out operation in RED. LEET's V1
resource allocator represents the external side with an explicit
`ExternalFrameResourceId` and descriptor in the request stream. The actual wgpu
handle is registered with the allocator before materialized resolve:

```rust
rctx.swap_external_texture(color, external_texture_id, external_texture_desc)?;
rctx.swap_external_buffer(lights, external_buffer_id, external_buffer_desc)?;

allocator.register_external_texture(external_texture_id, desc, texture, view)?;
allocator.register_external_buffer(external_buffer_id, desc, buffer)?;
```

This keeps request replay independent from raw GPU handle identity.

The semantic behavior is:

```text
PreConsume
  record SwapWithExternal(tag, external id + descriptor)
  resources are not resolved yet

Resolve
  attach the registered external handle to the planned external-swap allocation

Consume
  replay/validate SwapWithExternal(tag, external id + descriptor)
  make the external resource the tag's current resource
  resolve later getters for the tag to the external resource
```

The external handle must be valid and registered before materialized resolve.
The old resource attached to the tag is marked restricted/non-cacheable when
ownership cannot be proven safe. The newly introduced external resource is
tracked for graph lifetime, but it is not owned, recycled, destroyed, or
physically aliased by the wgpu backend.

RED's direct `RequestInjection` wrapper is texture-shaped in the inspected
allocator `.cpp`, while the public surface also declares external swap for both
textures and buffers. LEET should expose clean import/injection operations for
both resource kinds because buffers are V1 frame resources.

### Usage Flags

RED usage flags:

```cpp
RFUF_Read
RFUF_Write
RFUF_NoDiscard
```

Rust target:

```rust
ResourceUsage::READ
ResourceUsage::WRITE
ResourceUsage::NO_DISCARD
```

`READ` means the current use depends on existing contents. `WRITE` means the
current use writes the resource. `NO_DISCARD` is a per-use hint that the
allocator should not treat the first use as safely discardable.

`NO_DISCARD` is not cross-frame persistence. It does not mean the resource must
survive into the next frame. Cross-frame resources, such as TAA history,
previous depth, temporal denoiser history, exposure history, or persistent
scratch buffers, need an explicit owner outside the transient frame allocator.
That owner imports the resource into the frame graph when a node needs it.

```text
history resource owner
  -> owns texture/buffer across frames
  -> imports current/previous handle into this frame
  -> allocator tracks use inside this frame only
```

The transient allocator should therefore support import/injection, but ordinary
`declare_resource` allocations remain frame resources.

### Phases

RED exposes allocator phases:

```cpp
RFP_Startup
RFP_PreConsume
RFP_Consume
RFP_Cleanup
```

Rust should preserve the same state machine semantically:

```rust
enum ResourceAllocatorPhase {
    Startup,
    PreConsume,
    Resolve,
    Consume,
    Cleanup,
}
```

`Startup` verifies the previous frame was cleaned up. `PreConsume` records the
request stream while resources are unavailable. `Resolve` validates requests,
computes lifetimes, and assigns backing resources. `Consume` replays requests and
allows typed resource retrieval. `Cleanup` releases per-frame state and ages
cached backing allocations.

`SetPhase(Consume, builder)` in RED may trigger resolve and can use
`job::Builder`. In Rust, resolve should be represented explicitly even if the
implementation enters it from the transition between `PreConsume` and `Consume`.
The exact threading mechanism can be Rust-native, but the synchronization point
is essential.

### Resource Descriptors

RED uses one packed descriptor type, `SRenderFlowTargetDesc`, with a
`FlowResourceType` discriminator:

```cpp
FRT_Texture
FRT_Buffer
```

The same C++ struct stores either texture fields or buffer fields. That is an
implementation detail. Rust should use an enum:

```rust
enum FrameResourceDesc {
    Texture(FrameTextureDesc),
    Buffer(FrameBufferDesc),
}
```

Texture descriptors need to preserve the meaningful RED concepts:

- format
- width and height
- optional max width and max height for dynamic resolution style growth
- texture dimension, with array textures distinct from 3D textures
- depth or array/slice information
- mip count and optional max mip count
- sample/render/sampled/storage usage capability
- clear behavior and optimal clear value
- constrained backend policy flags when needed
- debug label

Buffer descriptors need to preserve:

- element count
- optional max element count
- element stride
- usage flags
- category or allocation class if LEET needs one
- optional typed-buffer format if LEET needs typed GPU buffer behavior
- debug label

RED stores some texture capabilities as negative flags, such as
`denyRenderTarget` and `denyUnorderedAccess`, plus a positive `isSamplable`.
Rust should prefer positive capability fields because wgpu usage flags are
positive:

```rust
struct FrameTextureCapabilities {
    render_attachment: bool,
    sampled: bool,
    storage: bool,
}
```

The descriptor must still produce legal wgpu usage flags. For example:

- render attachment maps to `TextureUsages::RENDER_ATTACHMENT`
- sampled maps to `TextureUsages::TEXTURE_BINDING`
- storage maps to `TextureUsages::STORAGE_BINDING`

RED rejects unsupported combinations in the mapper. LEET should do the same
with explicit validation errors. In particular, 3D depth frame textures should
be rejected unless a real supported use case appears, and 3D frame textures
should only expose usages that wgpu can create legally.

RED allows only known-safe extra backend flags. LEET should not expose an
unstructured `u64` bag of backend bits. Backend policy should be typed and
constrained:

```rust
struct FrameTextureBackendPolicy {
    // Fields here should be explicit LEET/backend decisions, not raw random bits.
}
```

RED has descriptor comparison modes for different purposes:

- exact equality
- equality ignoring maximum size
- compatibility for swap

Rust should keep those as explicit methods:

```rust
impl FrameResourceDesc {
    pub fn is_compatible_for_swap(&self, other: &Self) -> bool;
    pub fn is_equal_ignoring_max_size(&self, other: &Self) -> bool;
    pub fn is_exact_match(&self, other: &Self) -> bool;
}
```

RED's behavior should be preserved semantically:

- all three methods fail if the resource kind differs
- exact match compares the full creation-relevant descriptor
- equality ignoring max size compares the concrete/current descriptor while
  ignoring dynamic-resolution maximums
- swap compatibility is not exact equality; RED ignores optimized-clear metadata
  and masks selected backend usage-extension bits that do not affect whether two
  resources can exchange logical identity

Texture comparison fields:

| Field group | Exact | Ignore max size | Swap compatible |
| --- | --- | --- | --- |
| format | yes | yes | yes |
| current width/height/depth | yes | yes | yes |
| max width/max height | yes | no | yes |
| slices/layers | yes | yes | yes |
| mip count | yes | yes | yes |
| max mip count | yes | no | yes |
| sampled/render/storage capability | yes | yes | yes |
| extra usage flags | yes | yes | yes |
| selected backend usage-extension bits | yes | yes | masked |
| optimal clear value | yes | yes | ignored |

Buffer comparison fields:

| Field group | Exact | Ignore max size | Swap compatible |
| --- | --- | --- | --- |
| element count/current size | yes | yes | yes |
| max element count/max size | yes | no | yes |
| element stride | yes | yes | yes |
| category/allocation class | yes | yes | yes |
| usage flags | yes | yes | yes |

For Bevy/wgpu, the exact field names differ, but the same distinction should
hold. `is_exact_match` validates duplicate declarations. `is_equal_ignoring_max_size`
supports dynamic-resolution reuse decisions. `is_compatible_for_swap` validates
whether two logical resources may swap backing identity without breaking later
users.

RED's node context builds many descriptors from view state. For example, color,
depth, velocity, composition, screenshot, and final-target descriptors derive
their current and maximum sizes from the current camera/render context. LEET
should keep this as a node-context helper layer:

```rust
let color_desc = rctx.color_target_desc(format);
let depth_desc = rctx.depth_target_desc();
let velocity_desc = rctx.velocity_buffer_desc();
```

These helpers only create descriptors. They must not allocate resources or
become allocator identity. The allocator still sees an explicit
`FrameResourceDesc::Texture(...)` or `FrameResourceDesc::Buffer(...)` request.

### Purpose

The frame resource allocator manages transient GPU resources used by render graph
nodes while rendering a frame.

Examples:

- scene color
- depth prepass target
- G-buffer targets
- velocity buffer
- lighting accumulation targets
- SSAO scratch textures
- bloom ping-pong textures
- depth pyramid textures
- clustered light lists
- GPU culling output buffers
- per-frame indirect draw buffers
- temporary compute buffers
- readback or staging buffers whose lifetime is tied to the frame graph

These resources are graph working memory. They are not material assets, mesh
buffers, image assets, shader pipelines, bind groups, or scene-owned persistent
buffers.

The allocator answers this question:

> Given a logical frame resource name and a resource creation descriptor, which
> actual GPU texture or buffer should this frame use?

### Non-Goals

The frame resource allocator must not own or replace:

- Bevy `Image` asset loading
- `GpuImage` preparation
- `RenderAssets<GpuImage>`
- material buffers
- per-instance buffers
- mesh vertex or index buffers
- pipeline or shader binding layouts
- bind groups / descriptor sets
- long-lived render asset storage

Persistent resources may be used by render graph nodes, but they are not
allocated by this system.

### RED Behavior To Preserve Conceptually

RED's `CRenderFlowResourceAllocator` is a frame-local render resource planner.
It records allocation and usage requests from graph nodes, resolves lifetimes,
and maps logical render-flow tags to actual GPU resources.

Important RED concepts:

- resources are addressed by logical name tags
- logical names may be scoped by render-flow space, such as per-camera space
- resources are textures or buffers
- graph nodes request allocation, use begin, use end, free, swap, and injection
- pre-consume records intent before real resources are available
- consume uses the resolved GPU resources
- resource lifetimes are per-frame
- actual GPU allocations may be cached across frames
- non-overlapping lifetimes can reuse pooled resources; RED can additionally
  physically alias backing memory on lower-level backends

The Rust design should preserve the boundary and intent, not the exact C++
implementation shape.

### Bevy Boundary

Bevy already has separate resource paths that matter here.

Asset textures:

```text
Main App Image asset
  -> extracted/prepared into RenderApp
  -> GpuImage::prepare_asset
  -> RenderDevice::create_texture_with_data or create_texture
  -> RenderAssets<GpuImage>
```

Persistent/specialized buffers:

```text
RenderApp systems
  -> mesh allocator, material bind group allocators, uniform buffers,
     component buffers, phase instance buffers, or LEET-specific buffer owners
  -> RenderDevice::create_buffer or queue writes
```

These buffers are not frame resources. They are long-lived or subsystem-owned
GPU data stores.

Transient render textures:

```text
RenderApp system
  -> frame resource allocator
  -> RenderDevice::create_texture, imported Texture, or a LEET transient texture pool
```

Transient render buffers:

```text
RenderApp system
  -> frame resource allocator
  -> RenderDevice::create_buffer, imported Buffer, or a LEET transient buffer pool
```

LEET should use these boundaries instead of duplicating them.

The frame resource allocator should sit above Bevy's raw `RenderDevice`,
`Texture`, `TextureView`, and `Buffer` layer:

```text
LEET frame resource allocator
  logical texture name + descriptor + lifetime
    -> LEET transient texture policy
      -> Bevy RenderDevice
        -> wgpu texture

  logical buffer name + descriptor + lifetime
    -> LEET transient buffer policy
      -> Bevy RenderDevice
        -> wgpu buffer
```

For asset textures, render nodes should use `RenderAssets<GpuImage>` or another
asset bridge. They should not go through the frame resource allocator.

For persistent buffers, render nodes should use the subsystem that owns that
buffer and only pass the raw `Buffer` reference into binding code. They should
not re-declare that buffer as a transient frame resource unless it is explicitly
imported for graph lifetime tracking.

### Descriptor Meaning

In this document, descriptor means resource creation description, not Vulkan
descriptor sets.

Texture descriptor:

```text
size
format
dimension
mip count
sample count
usage flags
view formats
label
```

Buffer descriptor:

```text
size
usage flags
mapped-at-creation flag
label
```

The allocator needs descriptors so it can create or reuse compatible GPU
resources. It does not need shader binding indices, bind group layouts, material
layouts, or descriptor sets.

For buffers, usage flags carry the same kind of creation-time contract that
texture usage flags do. A temporary buffer that will be written by compute and
then consumed as indirect draw data needs those usages in the buffer descriptor.
The allocator should not infer shader bindings, but it must know the buffer was
created with legal GPU usages.

RED maps typed GPU buffers differently from structured buffers: typed buffers
derive byte size from element format, while structured buffers derive byte size
from element stride. LEET may not need typed frame buffers immediately, but the
descriptor should leave room for a buffer category/allocation class so that
typed, structured, indirect, staging, and readback policies do not collapse into
one ambiguous `size` field.

### Resource Kinds

The allocator owns transient textures and transient buffers from the first real
version.

```rust
FrameResourceDesc::Texture(...)
FrameResourceDesc::Buffer(...)
```

Textures are expected to be the first heavily used path, but buffers are part of
the same contract. RED uses the flow allocator for both render targets and
temporary GPU buffers, such as clustered/deferred lighting buffers and
post-process constant/working buffers. A texture-only design would force a later
API break.

The resource kind determines which descriptor, backing allocation, compatibility
checks, and resolved handle type are used.

```text
Texture tag -> texture descriptor -> resolved texture/view
Buffer tag  -> buffer descriptor  -> resolved buffer
```

The allocator should store raw Bevy/wgpu-facing resources or thin handles to
them, not typed higher-level LEET buffer abstractions.

Correct boundary:

```text
Texture / TextureView
Buffer
```

Incorrect boundary:

```text
RawArrayBuffer<T>
GpuOnlyBuffer<T>
MaterialBuffer<T>
InstanceBuffer<T>
```

Typed buffer wrappers may expose raw buffers to nodes, but their ownership and
update policy belongs to their own subsystem.

### V1 Contract

The first real allocator version must be compatible with the full RED-style
resource-flow model. It does not need every optimization to be perfect on day
one, but it must not expose a simplified API that prevents the final model from
being implemented later.

The rule for V1 is: implement the real shape now, while allowing conservative
policy tuning where the exact heuristic is not important yet. Future work should
extend prepared hooks and data fields, not rewrite the allocator around a
different model. Eviction and reuse heuristics may start conservative; the
`FrameResourcePool`, ownership tracking, lifetime model, request replay
validation, and reuse decision points must exist from the first implementation.

Required V1 behavior:

- logical resource names/tags
- render-flow spaces for per-camera or otherwise scoped resources
- texture and buffer resource descriptors
- texture and buffer declaration by descriptor
- texture and buffer declaration by another resource's descriptor
- imported/injected external resources
- recorded resource-existence queries through `is_declared`
- use-begin and use-end operations
- read/write/read-write usage flags
- explicit free/end lifetime operation
- swap between logical resources
- swap with an external texture or buffer reference
- two-phase pre-consume and consume protocol
- exact request-stream validation between pre-consume and consume
- per-frame lifetime analysis
- compatibility checks for descriptors
- retrieval only after resources have been resolved
- frame resource pool cache across frames
- lifetime-aware backing reuse for compatible non-overlapping lifetimes
- backend abstraction that keeps physical-memory-aliasing hooks possible for a
  future non-wgpu backend, without claiming that wgpu supports it
- debug dumps of logical resources, lifetimes, and backing resources

The allocator may internally take several implementation passes, but every pass
must keep this API direction intact. Unsupported operations should be absent or
fail loudly. They must not silently behave like a simpler cache.

### Execution Phases

The allocator has a strict phase model.

```text
Startup
  verify the previous frame was cleaned up

PreConsume
  execute graph/resource declarations
  record requests only
  actual GPU resources are not available

Resolve
  validate the recorded pre-consume request stream structurally
  compute lifetimes
  choose or create backing allocations
  assign logical resources to backing resources

Consume
  execute the graph for real
  replay allocation/use/free/swap/decision requests
  validate replay against the pre-consume request stream
  RTGet-style access returns actual texture/buffer handles

Cleanup
  release per-frame state
  age or evict cached backing allocations
```

Calling resource retrieval APIs during `PreConsume` is invalid. Calling
allocation or usage APIs during `Consume` must reproduce the same logical request
stream as `PreConsume`.

Request APIs are invalid during `Startup`, `Resolve`, and `Cleanup`. Typed
getters are invalid until `Consume`. These should be strict errors from V1:

```rust
ResourceAllocatorError::InvalidPhase {
    operation: "get_texture",
    phase: ResourceAllocatorPhase::PreConsume,
}
```

Strict phase checks prevent nodes from accessing unresolved resources, mutating
the stream outside the recording/replay phases, or hiding replay divergence.

### Request Model

The allocator records a per-frame request stream. The stream is the source of
truth for lifetime analysis.

Request kinds:

- declare texture by descriptor
- declare buffer by descriptor
- declare by descriptor copied from another tag
- inject external texture
- inject external buffer
- query whether a tag is allocated
- use begin
- use end
- free
- swap tags
- swap tag with external texture/buffer
- begin queue
- end queue
- queue sync
- decision

Rust should model requests as an enum with variant-specific payloads, not as a
packed C++ union:

```rust
enum ResourceRequest {
    Declare { tag: RenderFlowNameTag, desc: FrameResourceDesc },
    DeclareLike { dst: RenderFlowNameTag, src: RenderFlowNameTag },
    Import { tag: RenderFlowNameTag, resource: ImportedFrameResource },
    IsDeclared { tag: RenderFlowNameTag },
    UseBegin { tag: RenderFlowNameTag, usage: ResourceUsage },
    UseEnd { tag: RenderFlowNameTag },
    Free { tag: RenderFlowNameTag },
    Swap { a: RenderFlowNameTag, b: RenderFlowNameTag },
    SwapWithExternal { tag: RenderFlowNameTag, resource: ImportedFrameResource },
    BeginQueue { queue: RenderQueueKind },
    EndQueue,
    QueueSync { sync: QueueSyncKind },
    Decision { value: bool },
}
```

The flow group may be stored once as request-entry metadata instead of repeated
inside every variant:

```rust
struct RequestEntry {
    group: RenderFlowGroup,
    request: ResourceRequest,
}
```

This matches RED's behavior for requests like `QueueSync`, where the group is
the stream context rather than part of the variant payload.

The exact Rust enum names can change, but the important part is that each
operation that can affect resource lifetime or replay order is visible in the
stream.

`decision` exists because the graph may branch based on runtime state. A branch
taken during `PreConsume` must be replayed the same way during `Consume`, or the
request streams diverge and the allocator cannot safely resolve resources.

During `PreConsume`, RED stores each request in its flow group's ordered request
list and also stores a sortable key:

```text
(logical tag hash, flow group, request index within group)
```

Resolve sorts these keys so it can analyze all requests for a tag without
losing the original request time. LEET should preserve the same split:

- per-group request streams preserve replay order
- per-tag sorted keys drive lifetime analysis
- encoded request time is the pair `(flow_group, request_index)`

### Internal Request State

RED's internal header separates several concepts that should stay separate in
LEET:

- request groups: per-flow-group ordered request streams
- consume cursors: one cursor per group used to validate replay
- allocation requests: per-frame logical allocation intervals produced during
  resolve
- tag lifetimes: per-tag timelines that map request time to allocation request
- frame resource allocations: actual GPU resources kept in a pool across frames

In Rust this points to shapes like:

```rust
#[derive(Clone, Copy, Eq, PartialEq, Ord, PartialOrd, Hash)]
struct RequestTime {
    group: RenderFlowGroup,
    index_in_group: u16,
}

#[derive(Clone, Copy, Eq, PartialEq, Ord, PartialOrd)]
struct PerTagRequestKey {
    tag_hash: u32,
    time: RequestTime,
}

struct RequestGroup {
    requests: Vec<ResourceRequest>,
    consume_cursor: usize,
    touched: bool,
}

struct TagLifetimeEvent {
    time: RequestTime,
    request_kind: ResourceRequestKind,
    allocation_request: AllocationRequestId,
}

struct AllocationRequest {
    tag: RenderFlowNameTag,
    desc: FrameResourceDesc,
    lifetime: RequestRange,
    unmodified_end: RequestTime,
    can_reuse_backing: bool,
    can_cache: bool,
    assigned: Option<FrameResourceAllocationId>,
}
```

`PreConsume` appends requests to the current group's stream. `Consume` advances
the group's consume cursor and validates that the incoming request exactly
matches the recorded request at that cursor. `Cleanup` must assert that every
touched group consumed all recorded requests.

Typed getters resolve against the current consume position. RED uses the current
group's consume cursor, effectively:

```text
RequestTime {
  group: current_flow_group,
  index_in_group: current_consume_cursor - 1
}
```

LEET should preserve that semantic. `get_texture(tag)` and `get_buffer(tag)` do
not resolve from the tag alone; they resolve from `(tag, current request time)`.

Nameless/autogenerated tags are reproduced from the same consume cursor. RED
builds the autogenerated id from the flow group and request index, then uses a
reserved flow-space value because the generated id is already unique. LEET does
not need the exact bit split, but it must assert that:

- nameless tags are generated from request-stream position, not from a global
  counter that can diverge between phases
- the generated id is never zero, so it cannot collide with ordinary named tags
- the same pre-consume and consume request position produces the same tag

The generated temp tag's string is diagnostic only. Multiple temp resources may
share the same debug name without becoming the same logical resource. Temp tags
are useful for per-node or per-pass scratch resources such as bloom intermediates,
SSAO scratch textures, depth pyramid temporaries, GPU culling scratch buffers,
and compute histogram buffers.

RED has hard static limits for group count, requests, and allocation requests.
LEET should not expose those exact limits as the API contract. It may reserve
capacity or use configurable limits internally, but overflow should fail loudly
with diagnostics.

### Frame Resource Pool

RED stores actual GPU objects in `SAllocation` entries owned by
`SAllocationPool`. An `SAllocation` contains a texture/buffer union plus one
`SRenderFlowTargetDesc`; RED uses the descriptor's resource kind to interpret the
stored GPU handle. The pool stores allocations through owning pointers because
allocation requests and tag lifetime events keep pointers back to the selected
allocation.

The RED header describes the same split explicitly:

```cpp
struct SAllocation {
    Uint64 m_startTime;
    Uint64 m_endTime;
    Bool m_usedThisFrame;
    union { TextureRef m_texRef; BufferRef m_buffRef; ResourceRef m_resRef; };
    SRenderFlowTargetDesc m_desc;
    Bool m_cacheble;
    Uint32 m_age;
};

struct SAllocationRequest {
    Uint64 m_startTime;
    Uint64 m_endTime;
    Uint64 m_endTimeUnmodified;
    SRenderFlowTargetDesc m_desc;
    Bool m_canBeAliased;
    Bool m_canBeCached;
    SAllocation* m_allocIndirection;
};
```

The important boundary is that `SAllocationRequest` is a per-frame plan, while
`SAllocation` is the actual pooled GPU resource that can survive across frames.
LEET should preserve that boundary with ids instead of raw pointers.

Rust should keep the same separation without raw pointers:

```rust
struct FrameResourcePool {
    allocations: Vec<FrameResourceAllocation>,
}

struct FrameResourceAllocationId(u32);

struct FrameResourceAllocation {
    desc: FrameResourceDesc,
    resource: FrameResource,
    concrete_size: FrameResourceConcreteSize,
    lifetime_this_frame: Option<RequestRange>,
    used_this_frame: bool,
    cacheable: bool,
    ownership: FrameResourceOwnership,
    estimated_bytes: u64,
    age: u32,
}

enum FrameResource {
    Texture(FrameTextureResource),
    Buffer(FrameBufferResource),
}

struct FrameTextureResource {
    texture: Texture,
    default_view: TextureView,
}

struct FrameBufferResource {
    buffer: Buffer,
}

enum FrameResourceOwnership {
    Owned,
    Imported,
    ExternalSwap,
}
```

The actual GPU resources live inside `FrameResourceAllocation`. The ID is only a
stable reference into the pool, used by allocation requests and tag timelines:

```text
tag lifetime event
  -> allocation request id
    -> frame resource allocation id
      -> FrameResourceAllocation in FrameResourcePool
```

Generic allocator storage should use `FrameResourceDesc`, not a typed descriptor,
so the resource kind stays explicit at the same boundary as the public API. A
typed texture or buffer pool may exist internally for faster scans, but the
cross-kind request, timeline, and resolve layers should keep the
`FrameResourceDesc` enum boundary.

`FrameResourceOwnership` is allocator policy metadata. It is not a replacement
for Rust borrowing. It tells cleanup, cache, and reuse code whether the allocator
created the resource and may recycle or destroy it:

- `Owned`: created by the frame resource pool; may be cached, reused, or evicted
  according to allocator policy
- `Imported`: externally owned handle borrowed for graph lifetime tracking; must
  not be destroyed, recycled, or evicted by the allocator
- `ExternalSwap`: handle involved in a swap with external ownership; reuse is
  disabled unless the operation explicitly transfers ownership back to the pool

Bevy/wgpu `Texture`, `TextureView`, and `Buffer` values are handle objects that
can be stored in these records. Ownership here is about allocator authority over
lifetime policy, not whether Rust can hold the handle value.

For wgpu, LEET does not need RED's physical `SAliasedBlock` structure. The pool
may store owned resources in a flat vector, split texture/buffer vectors, or a
generational arena. The public contract is that IDs remain valid for resolved
assignments until cleanup, and stale IDs are caught loudly in diagnostics.

RED drops allocation requests that were declared but never used. LEET should do
the same: `declare_resource` records descriptor intent, but an owned GPU object
does not need to be created if the resolved lifetime has no real use.

### Lifetime And Pool Reuse

Each logical resource tag has a per-frame lifetime. Lifetimes are derived from
allocation, use-begin, use-end, free, swap, and imported-resource events.

The allocator must determine:

- first request that makes a resource alive
- last request that touches the resource
- read/write intent during each use range
- whether a resource can be cached
- whether a resource can reuse a pooled backing allocation
- which backing allocation currently belongs to a logical tag
- whether the tag resolves to a texture or a buffer

Tag lifetime is time-indexed. A tag is not just:

```text
tag -> backing resource
```

It is:

```text
(tag, request time) -> backing resource
```

The trivial case can use a fast path where a tag has one allocation for the
whole frame. Non-trivial cases, such as `swap`, `swap_with_external`, import, or
declaration by another tag, need a timeline of important events. `get_texture`
and `get_buffer` therefore resolve the resource for the current consume cursor,
not merely the tag name.

RED classifies tags into two paths during resolve:

- trivial tags: declared by descriptor and only used/freed without remapping;
  these get one allocation request and a fast-path tag lifetime
- non-trivial tags: touched by `swap`, `swap_with_external`, or
  `declare_resource_like`; these are replayed in request-time order with a
  current tag-to-allocation mapping and per-tag timeline events

This distinction is an implementation optimization, not a public behavior
difference. LEET can use the same fast path, but correctness must come from the
timeline model.

RED's resolve implementation gives the concrete model to preserve:

```cpp
LocalTagTracingStruct currentTagToAllocReqMapping;
```

That map is the current timeline state for non-trivial tags:

```text
tag -> currently active allocation request
```

`Swap` updates that map, closes both current lifetimes at the swap timestamp,
and records timeline events for both tags. `SwapWithExternal` closes the old
allocator-owned allocation and marks it unsafe for normal reuse:

```cpp
allocReqA->m_canBeAliased = false;
allocReqA->m_canBeCached = false;
allocReqA->m_endTime = nonTrivKey;
```

In LEET/wgpu terms, this does not mean physical aliasing. It means the old owned
resource must not be returned to the ordinary reusable pool unless ownership and
future lifetime are explicitly safe. The external resource becomes the current
resource for that tag through allocation indirection/timeline state.

`declare_resource_like` mirrors RED's `AllocByTag`: copy the source descriptor
from either the trivial lifetime fast path or the current non-trivial mapping.
RED then strips ESRAM placement:

```cpp
copyDesc.m_extraUsageFlags &= ~GpuApi::TEXUSAGE_ESRAMResident;
```

LEET should apply the same principle: copy descriptor shape, kind, size,
format, usage/capabilities, and clear behavior, but do not accidentally inherit
backend placement policy that should be chosen for the destination resource.

Declared-but-unused resources do not need GPU storage. RED records the request,
but only creates an allocation request if a later timestamp actually touches the
resource after allocation:

```cpp
if (startTime != -1 && endTime != 0) {
    InitAllocRequest(allocReq, startTime, endTime, *targetDesc);
}
```

LEET should preserve this behavior. `declare_resource` records intent and
participates in replay validation, but an owned texture or buffer is only
assigned when the resolved lifetime has a real use.

Use ranges are normally derived from `use_begin` and `use_end`. RED also allows
use requests for a tag that was never allocated in that replay path to be skipped
during non-trivial lifetime solving, so optional resource code can stay readable.
LEET may support the same behavior only for explicit optional/null flows; ordinary
uses of undeclared tags should fail loudly.

Both `use_begin` and `use_end` count as lifetime touches in RED's resolve path.
For the first use, RED sets the allocation start time; every later use request
updates the end time:

```cpp
if (allocReq->m_startTime == 0) {
    allocReq->m_startTime = nonTrivKey;
}
allocReq->m_endTime = nonTrivKey;
```

LEET should still validate begin/end balance, but lifetime coverage should
include both endpoints.

Queue fork/join events can extend lifetimes. RED stores fork/join request times,
then extends allocation end times when an allocation starts inside an async
compute interval. LEET should model queue constraints as lifetime extension
inputs before assigning pooled resources.

The RED rule is conservative:

```cpp
if (start > forkTime && start < joinTime) {
    end = Max(end, joinTime);
}
```

LEET should keep the same correctness rule for reuse safety. On wgpu this only
prevents premature reuse; it does not create alias barriers.

Resource lifetimes should also have internal start/end hooks. On RED these emit
physical aliasing barriers and retire operations. On LEET's wgpu backend they
are correctness and diagnostics hooks, not physical aliasing:

```rust
fn on_lifetime_start(allocation: FrameResourceAllocationId, usage: ResourceUsage) {
    // assert allocation is not already live for an overlapping logical resource
    // record discard/preserve intent from NO_DISCARD
}

fn on_lifetime_end(allocation: FrameResourceAllocationId) {
    // mark the pooled allocation available for later compatible reuse
}
```

These hooks protect reuse correctness even when the backend cannot alias memory.
They are internal allocator/backend hooks, not public node-facing API.

On RED, two resources may physically alias backing memory only if:

- their descriptors are compatible for aliasing
- they are the same resource kind, texture with texture or buffer with buffer
- their lifetimes do not overlap
- neither resource was marked non-aliased by import/swap behavior
- required queue/fork/join constraints do not extend the lifetime overlap

LEET's V1 wgpu backend does not support physical memory aliasing. It must not
claim to create multiple live wgpu resources over the same memory block, and it
does not expose aliasing barriers. The V1 contract is lifetime-aware backing
reuse:

- two overlapping logical lifetimes must never be assigned the same owned wgpu
  texture or buffer
- a pooled backing allocation may be reused for another logical resource only
  after the previous logical lifetime is over
- reuse requires the same resource kind and compatible descriptor
- allocation requests should be considered in descending size and then stable
  start-time order, so large resources find capacity first and reuse is stable
  across frames
- imported/external resources are tracked for graph lifetime, but are not owned,
  recycled, or destroyed by the allocator
- swap-with-external can mark the old pooled resource non-cacheable or
  non-reusable when ownership cannot be proven safe

RED has two assignment paths:

- aliasable requests are placed into aliased blocks when type, capacity, and
  lifetime ranges allow it
- non-aliasable requests, such as resources affected by external swap, use a
  stricter cache path: exact descriptor match, no overlapping use, and the
  candidate must remain cacheable

LEET/wgpu should translate this into two reuse classes rather than RED's
physical aliasing blocks. Ordinary transient resources can use capacity-based
pooled reuse. External/import/swap-affected resources use stricter reuse or no
reuse, depending on ownership and cacheability.

The core RED overlap test treats touching endpoints as overlap:

```cpp
return request->m_endTime >= spanStart && spanEnd >= request->m_startTime;
```

LEET should use the same conservative rule unless it deliberately changes the
entire allocator to half-open ranges. The default V1 rule is therefore:

```rust
fn lifetimes_overlap(a: RequestRange, b: RequestRange) -> bool {
    a.end >= b.start && b.end >= a.start
}

fn may_reuse_owned_allocation(
    old: &FrameResourceAllocation,
    request: &AllocationRequest,
) -> bool {
    old.ownership == FrameResourceOwnership::Owned
        && old.cacheable
        && old.desc.can_reuse_for(&request.desc)
        && match old.lifetime_this_frame {
            Some(old_lifetime) => {
                !lifetimes_overlap(old_lifetime, request.lifetime)
                    && old_lifetime.end < request.lifetime.start
            }
            None => true,
        }
}
```

Same-frame reuse is part of V1. A pooled owned wgpu texture or buffer may be
assigned to another logical resource in the same frame only after the previous
logical lifetime is completely over, after queue/fork/join lifetime extension,
and only if descriptor/capacity checks pass.

RED's physical aliasing path has an extra detail:

```cpp
!aliasedResource->m_usedThisFrame
    && aliasedResource->m_desc.IsEqualIgnoringMaxSize(allocRequest->m_desc)
```

That avoids reusing the same RED resource reference twice inside one frame,
because RED can create multiple resource objects over one aliasing block. LEET
does not have physical aliasing on wgpu, so this exact restriction should not be
copied as "no same-frame reuse." Instead, LEET must enforce the stricter
timeline and ownership checks above for actual handle reuse.

RED's non-aliasable path is closer to LEET/wgpu handle reuse:

```cpp
!RequestOverlapsSpan(allocReq, old.m_startTime, old.m_endTime)
    && allocReq->m_startTime > old.m_endTime
```

The second condition matters. It proves the old use happened earlier on the
request timeline, not merely that two intervals failed to overlap due to an
edge-case. The RED comment calls out `txaa accum for mirror + camera case`; LEET
should keep this strict ordering for external/swap-affected resources and for
any same-frame handle reuse where queue ordering is not trivial.

RED sorts frame allocation requests before pool assignment:

```cpp
return request[a].m_size > request[b].m_size;
return request[a].m_startTime < request[b].m_startTime;
```

LEET should preserve that policy: place large requests first, then use start
time as a stable tie-breaker. This is a pool assignment policy, not a public API
rule, but it affects reuse quality and diagnostics.

The allocator should still keep the RED-style lifetime/reuse decision model so a
future backend with placed resources can add true physical aliasing without
changing the public API.

### Dynamic Resolution

RED descriptors carry both current size and maximum size. RED uses the current
size for ordinary concrete resources and the maximum size for physical aliasing
blocks. LEET/wgpu should preserve the semantic split without claiming physical
aliasing support.

Texture descriptors should represent:

```rust
struct FrameTextureDesc {
    descriptor: TextureDescriptor<'static>,
    current_size: Extent3d,
    max_size: Option<Extent3d>,
    current_mip_level_count: u32,
    max_mip_level_count: Option<u32>,
    clear_to_zero: bool,
    optimal_clear_value: Option<[f32; 4]>,
}
```

Buffer descriptors should represent:

```rust
struct FrameBufferDesc {
    descriptor: BufferDescriptor<'static>,
    current_size_bytes: u64,
    max_size_bytes: Option<u64>,
    element_stride: Option<u64>,
}
```

Additional capability/category/backend-policy fields may be added when a real
backend need appears, but they are not required to preserve the current RED
allocator contract. The important V1 rule is that current size and optional
maximum capacity are explicit LEET fields, while the embedded wgpu descriptors
remain templates for creation-time non-size fields.

The explicit size fields are the source of truth for frame-resource sizing. The
embedded wgpu descriptors are templates for non-size creation fields such as
format, dimension, usage, sample count, mapped-at-creation, and label. When LEET
creates a concrete wgpu resource, it must build the final descriptor from the
selected allocation shape:

```text
current_allocation_shape -> concrete TextureDescriptor.size / BufferDescriptor.size
max_capacity_shape       -> pool capacity comparison
```

This avoids two independent size authorities inside `FrameTextureDesc` or
`FrameBufferDesc`.

Validation rules:

- `max_size` must be greater than or equal to `current_size`
- `max_mip_count` must be greater than or equal to `current_mip_count`
- `max_size_bytes` must be greater than or equal to `current_size_bytes`

wgpu receives one concrete allocation size:

```text
TextureDescriptor.size = concrete backing texture size
BufferDescriptor.size  = concrete backing buffer size
```

LEET/wgpu policy:

- the logical descriptor records the current size used by this frame
- the maximum size is a legal growth bound and compatibility/capacity signal
- backing allocation size must be at least the current size
- a cached backing can be reused if it is large enough for the current size and
  the non-size descriptor fields are compatible
- if current size grows beyond the cached backing, allocate a larger backing
- if current size shrinks, the larger backing may stay cached until eviction
- the allocator must never treat resources in different flow spaces as the same
  logical resource, even if they share a debug name such as `scene_color`

This supports dynamic resolution without forcing every wgpu resource to be
allocated at maximum size on first use.

The allocator should expose descriptor shape helpers internally:

```rust
impl FrameResourceDesc {
    fn current_allocation_shape(&self) -> FrameResourceShape;
    fn max_capacity_shape(&self) -> FrameResourceShape;
}
```

RED's `Desc::Map(..., aliasingBlock = false)` corresponds to the current
allocation shape. `Desc::Map(..., aliasingBlock = true)` corresponds to the
maximum capacity shape. LEET uses the same distinction for pooled reuse even
though wgpu does not expose physical aliasing.

### Cache Eviction

The `FrameResourcePool` and eviction mechanism are V1 behavior, not a later rewrite.
The initial policy may be age-based, but the allocator must already track the
fields needed for richer policies.

Each owned `FrameResourceAllocation` should track:

- resource kind and descriptor
- concrete backing size and estimated bytes
- whether it was used this frame
- whether it is cacheable
- ownership/import status
- age in unused frames
- age in overcapacity frames, when a larger cached owned allocation keeps being
  reused for smaller current-size requests

Cleanup updates cache state:

```text
if allocation was used this frame:
  age = 0
else:
  age += 1

if process_eviction && cacheable && age > eviction_threshold:
  destroy owned frame resource
```

Imported or external resources are never evicted by the frame allocator because
the allocator does not own them. `clear_all_caches` should exist for debugging,
device-pressure handling, and explicit teardown, and it must only destroy owned
cached frame resource allocations.

RED uses an unused-frame threshold of 8 during cleanup. LEET's initial V1 pool
default is `FrameResourcePool::DEFAULT_MAX_UNUSED_AGE = 3`; it is configurable,
explicit in code, and visible in diagnostics.
`clear_all_caches` should only be valid during cleanup or another quiescent
phase where no resolved frame resources are live.

RED cleanup separates tracking reset from eviction:

```cpp
if (resource->m_usedThisFrame) resource->m_age = 0;
else if (advanceAge) resource->m_age++;

resource->m_startTime = 0;
resource->m_endTime = 0;
resource->m_usedThisFrame = false;
```

Non-cacheable resources are removed from the allocator cache, while old
cacheable resources are released after the age threshold:

```cpp
if (!cachedRes->m_cacheble) {
    remove_from_pool();
} else if (processEviction && cachedRes->m_age >= ageThreshold) {
    release_gpu_resource();
    remove_from_pool();
}
```

LEET should mirror the policy with ownership-aware actions:

```rust
match allocation.ownership {
    FrameResourceOwnership::Owned if !allocation.cacheable => {
        pool.destroy_owned(allocation_id);
    }
    FrameResourceOwnership::Owned if allocation.age >= eviction_threshold => {
        pool.destroy_owned(allocation_id);
    }
    FrameResourceOwnership::Imported | FrameResourceOwnership::ExternalSwap => {
        pool.forget_external(allocation_id);
    }
    _ => {
        allocation.reset_frame_tracking();
    }
}
```

For wgpu, dropping an owned `Texture` or `Buffer` handle is the release action.
For imported/external resources, cleanup must only forget the allocator record;
the allocator does not own the underlying GPU object.

RED also ages oversized aliasing blocks differently: if a block was used but not
filled near its maximum size for several frames, it can be evicted so memory can
shrink after dynamic resolution drops. LEET preserves the policy concept as
overcapacity eviction:

```text
if cached allocation concrete size is much larger than recent requested size
and it stays overcapacity for eviction_threshold frames:
  evict the owned cached allocation
```

V1 tracks this as `FrameResourceAllocation::overcapacity_age`. When an owned
cached allocation is reused for a smaller current allocation shape, the counter
ages. When the allocation is used at its concrete capacity again, the counter
resets. Cleanup can evict owned cache entries whose overcapacity age exceeds the
pool threshold, so peak dynamic-resolution memory does not have to stay alive
forever in a steady lower-resolution scene. The exact ratio can become a tunable
policy later without changing the public allocator model.

### Phase Replay And Retrieval

RED's final phase/retrieval implementation reinforces the strict state machine:

```cpp
case RFP_Startup:    assert(previous == RFP_Cleanup)
case RFP_PreConsume: assert(previous == RFP_Startup)
case RFP_Consume:    assert(previous == RFP_PreConsume); Resolve(builder)
case RFP_Cleanup:    assert(previous == RFP_Consume)
```

LEET should preserve the same transition graph. If Rust exposes `Resolve` as an
explicit phase, entering `Resolve` should still only be valid after
`PreConsume`, and entering `Consume` should only be valid after successful
resolve.

Cleanup must prove that consume replay reached the end of every touched request
group. RED checks:

```cpp
RED_ASSERT(group.m_numConsumed == group.m_requestsPerGroup.Size());
```

This catches a graph that replayed matching requests at the beginning but skipped
some trailing requests during `Consume`. LEET should make this a normal
validation error in debug/development builds and a loud allocator error in
strict validation mode.

Cleanup then resets per-frame state:

```text
tag lifetimes
lifetime events
injected allocation records
per-group request streams and consume cursors
frame allocation requests
request counters
non-trivial tag markers
fork/join request markers
```

After per-frame state is reset, cleanup ages or evicts cached pool allocations.
RED uses an age threshold of 8 frames:

```cpp
const Uint32 ageThreshold = 8;
ReleaseCachedAllocs(1, ageThreshold);
```

LEET's current V1 default is lower (`DEFAULT_MAX_UNUSED_AGE = 3`) so tests and
early renderer development expose eviction behavior quickly. The threshold is a
pool construction parameter and can be tuned later without changing the public
allocator model.

Typed retrieval is only valid during `Consume`. RED resolves getters from the
current flow group and the last consumed request index:

```cpp
const Uint32 reqId = Max<Uint32>(1, group.m_numConsumed) - 1;
GetResourceAtGivenTime(EncodeReqId(flowGroup, reqId));
```

LEET should mirror this with `RequestTime { group, index_in_group }` derived
from the current consume cursor. A getter must never resolve from the tag alone.

Trivial tag lifetimes can skip timeline lookup:

```cpp
if (auto ptr = tagLifetime.m_pAllocReqTrivial) {
    return ptr->m_allocIndirection->m_texRef;
}
```

Non-trivial tag lifetimes use sparse timeline events. RED does not record every
request time; it finds the event interval that covers the current consume time:

```cpp
if (event.timestamp >= current_time) {
    finalIndex = i == 0 ? 0 : i - 1;
}
```

LEET's `TagLifetime::Timeline` should preserve this sparse-event model. Timeline
events mark important changes such as use, swap, and external swap. Getters ask
which allocation request is current at the current consume cursor.

Optional retrieval must be explicit. RED's getter has an `allowNull` flag:

```cpp
RED_FATAL_ASSERT(allowNull, "Texture not found and never declared");
```

Rust should expose that distinction directly:

```rust
rctx.get_texture(tag) -> Result<FrameTextureHandle>
rctx.try_get_texture(tag) -> Result<Option<FrameTextureHandle>>

rctx.get_buffer(tag) -> Result<FrameBufferHandle>
rctx.try_get_buffer(tag) -> Result<Option<FrameBufferHandle>>
```

`clear_all_caches` is also phase constrained. RED only allows it during cleanup:

```cpp
RED_ASSERT(m_phase == RFP_Cleanup);
ReleaseCachedAllocs(0, 0);
```

LEET should allow cache clearing only during `Cleanup` or an explicitly
quiescent/device-lost path where no resolved frame resources are live.

### Frame Resource Pool Sources

The allocator should use Bevy where it is the correct backend, but LEET owns the
transient frame-resource policy.

Texture backing options:

- direct `RenderDevice::create_texture`
- imported `Texture`/`TextureView` for external resources
- LEET transient texture pool for repeated scratch-texture allocations

Buffer backing options:

- direct `RenderDevice::create_buffer`
- imported `Buffer` for external resources
- LEET transient buffer pool for repeated scratch-buffer allocations
- optional staging/readback buffer handling when a graph resource explicitly
  needs CPU transfer

The public allocator API should be independent of the chosen backing source.
Nodes should see logical tags and resolved texture/buffer handles, not whether a
resource came from direct creation, a LEET pool, or import.

Texture pool reuse and buffer pool reuse are different backend policies. The
allocator contract should make them look uniform to nodes.

### Backend Policy Notes

RED's allocator `.cpp` has implementation TODOs for alignment, ESRAM, defrag,
brute-force optimization, and deferring actual allocation until first use. These
are backend policy concerns, not node-facing API concerns.

LEET should handle them as follows:

- alignment is part of the resource creation/reuse policy and must be respected
  by the pool when matching or creating textures and buffers
- console-specific memory flags such as ESRAM are not part of the LEET public
  contract; backend-specific usage bits may exist behind `FrameResourceDesc`
- defrag is not a V1 requirement for wgpu, but pool storage should not make
  future compaction or migration impossible
- allocation may be implemented with conservative scans first, but the data
  model must still be request/lifetime/pool based
- do not defer creation to first use in a way that bypasses resolve; if lazy
  physical creation is added later, resolve must still produce stable validated
  assignments before consume retrieval

### Implementation Passes

The allocator can still be built in passes, but these are implementation slices,
not simplified product versions.

Pass 1: Types and phase shell

- define tags, flow spaces, request types, descriptors, usage flags, and phases
- reject invalid phase operations loudly
- store request streams for texture and buffer operations

Pass 2: Request validation

- validate pre-consume/consume request matching
- validate descriptor compatibility
- validate use begin/end balance
- validate retrieval only after resolve

Pass 3: Lifetime analysis

- derive per-tag lifetime intervals
- model swap/import events
- produce debug dumps for requests and lifetimes

Pass 4: Pool assignment

- assign textures and buffers to `FrameResourcePool` allocations
- cache owned pool allocations across frames
- support imported resources without taking ownership incorrectly

Pass 5: Pool reuse

- assign compatible non-overlapping resources to reused pool allocations
- keep conservative lifetime extension where queue/fork/join behavior requires it
- expose diagnostics for reuse decisions

No pass may introduce a public API that assumes the allocator is only:

```text
logical name -> descriptor -> cached texture
```

That shape is useful as an internal fallback, but it is not the allocator
contract.

The same warning applies to buffers. A public API that assumes buffers are always
owned by specialized systems would be incompatible with RED-style temporary graph
buffers.

### Must Not Duplicate Bevy Memory

The allocator must avoid allocating the same logical render target that Bevy
already prepared.

Two valid modes:

- Keep Bevy core view target systems and use their prepared view targets.
- Replace Bevy core view target systems and let LEET allocate equivalent targets.

Invalid mode:

- Bevy allocates a main view target and LEET allocates another equivalent main
  view target for the same camera path without an explicit reason.

### Open Questions

- Will LEET keep Bevy's `prepare_view_targets`, or replace it?
- Should the first allocator live as a render-world resource, or be rebuilt as a
  frame-local object passed through graph execution?
- Which graph nodes should first exercise transient LEET textures and transient
  LEET buffers?
- What initial buffer pool policy should LEET use before adding memory-budget
  pressure heuristics?
- How should external resources be named when imported into the graph?
- Which first multi-camera path should exercise per-camera flow spaces?
